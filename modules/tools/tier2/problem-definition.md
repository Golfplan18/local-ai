# Problem Definition Question Bank

*Tier 2 module. Loads into the Breadth model context window as a complete unit.*

## Trigger Conditions

- Phase B or Phase C processing
- Any session where problem definition work is explicitly requested
- Triage gate assigns Tier 2 or Tier 3

## Purpose

This module provides structured questions that surface hidden problem structure. It runs alongside Tier 1 tools (especially Concept Fan, CAF, and Challenge) to ensure the problem definition process does not settle prematurely on the first plausible framing.

---

## Category 1: Problem Boundary Questions

These determine what is inside and outside the problem scope.

1. What would a complete solution to this problem actually look like? What would be different in the world?
2. Who experiences this as a problem? Is there anyone for whom the current situation is working well?
3. Where does this problem stop and an adjacent problem begin? Are we looking at one problem or several problems wearing the same label?
4. What is the smallest version of this problem that still captures its essential character?
5. If this problem were solved perfectly, what new problems would that create?

## Category 2: Problem History Questions

These reveal how the problem arrived at its current state.

6. When did this become a problem? What changed?
7. Has this problem been solved before — here or elsewhere? What happened to those solutions?
8. What previous attempts to solve this have been made, and why did they fail or produce partial results?
9. Is this problem getting worse, getting better, or staying stable? What is the trajectory?
10. Is this a new problem, a recurring problem, or a permanent condition being treated as a problem?

## Category 3: Problem Ownership Questions

These identify who holds the problem and who holds the levers.

11. Who has the authority to implement a solution? Is the person asking also the person who can act?
12. Who would need to agree for a solution to be implemented? What is the approval chain?
13. Who benefits from this problem remaining unsolved?
14. What resources (time, money, attention, political capital) are actually available for solving this?
15. Is the person presenting the problem the person who experiences it, or are they reporting it on behalf of someone else?

## Category 4: Problem Framing Questions

These test whether the stated framing is the right one.

16. Is this a problem of too much or too little? (Resources, information, options, constraints, time)
17. Is this a problem of knowing what to do, or a problem of doing what is already known?
18. Is the stated problem a symptom of a deeper problem? What would that deeper problem be?
19. What assumption, if removed, would dissolve this problem entirely?
20. Is this problem caused by the absence of something, or the presence of something?

## Category 5: Solution Space Questions

These map what kinds of solutions are available and what constrains them.

21. What constraints on the solution are genuine physical or logical constraints, and which are conventions or policies?
22. What is the cost of doing nothing? Is inaction a viable option?
23. Is the user looking for the best solution, a good-enough solution, or the fastest solution? Do they know which one they want?
24. What has been ruled out already, and why? Are those exclusions still valid?
25. Is there a deadline driving this, and is the deadline real or artificial?

## Category 6: Problem Definition Quality Check

Run these after a candidate problem definition has emerged.

26. Can we state the problem in one sentence without using jargon or abstractions? If not, the definition is not yet sharp enough.
27. Would two different people reading this problem definition attempt to solve the same problem?
28. Does the problem definition suggest a direction for solution, or does it remain open to multiple solution approaches?
29. Does the problem definition contain a hidden solution? (e.g., "We need a better X" presupposes X is the right approach.)
30. Has the problem definition been tested against the original prompt? Does it still address what the user actually asked about?

---

## Integration Notes

- In Phase B, questions from Categories 1-4 feed into Concept Fan and Challenge operations.
- In Phase C, questions from Categories 4-6 serve as quality checks on the emerging problem definition.
- Category 3 (Problem Ownership) surfaces information that OPV and FGL need but may not generate on their own.
- Category 5 maps directly to the constraint identification work that precedes mode selection.
