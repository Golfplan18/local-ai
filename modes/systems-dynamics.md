# MODE: Systems Dynamics

## TRIGGER CONDITIONS

Positive triggers: The problem involves feedback loops, unintended consequences, or counterintuitive behavior. Interventions keep producing unexpected results. The situation involves delays between action and effect. Multiple variables interact in circular rather than linear ways. "Why does fixing X make things worse," "what are the leverage points," "this keeps happening despite our attempts to fix it."

Negative triggers: IF the relationships are static and structural without temporal dynamics, THEN route to Relationship Mapping. IF the user wants to trace a specific failure backward through a causal chain, THEN route to Root Cause Analysis. IF the user is mapping territory rather than analyzing system behavior, THEN route to Terrain Mapping.

## EPISTEMOLOGICAL POSTURE

Causation is circular, emergent, and feedback-driven, not linear. Behavior arises from structure, not from individual intentions or single causes. Systems exhibit counterintuitive behavior — corrective programs are often counteracted by the system's own feedback dynamics. The observer is part of the system. Understanding requires identifying not just connections but the type of connection: reinforcing loops (R), balancing loops (B), delays, stocks, and flows. Leverage points exist but are often non-obvious — the most effective interventions are rarely the most obvious ones.

## DEFAULT GEAR

Gear 4. Independent structural analysis is the minimum. Two models may identify different feedback loops, different leverage points, and different system archetypes. Convergence increases confidence. Divergence reveals structural features worth examining that anchored analysis would suppress.

## RAG PROFILE

Retrieve systems thinking literature, system dynamics case studies, system archetype libraries, and domain-specific systems analyses. Retrieve Donella Meadows' leverage points framework. IF the domain has known system dynamics models, retrieve those. Deprioritize linear causal analyses and single-factor explanations.

## DEPTH MODEL INSTRUCTIONS

White Hat directives:
1. Identify stocks (accumulations) and flows (rates of change). What is accumulating, and what drives the rate of accumulation and depletion?
2. Identify all feedback loops and classify each as reinforcing (R — amplifies change) or balancing (B — resists change).
3. Identify delays — where the effect of an action takes time to materialize. Delays are the most common source of counterintuitive system behavior.

Black Hat directives:
1. For each feedback loop, assess whether it is genuinely circular or a linear chain misidentified as circular.
2. Identify at minimum one counterintuitive behavior — a case where intervention produces the opposite of the intended effect due to feedback dynamics.
3. Challenge the Breadth model's leverage point identification. Are proposed points genuinely high-leverage (affecting structure or goals) or low-leverage (adjusting parameters within existing structure)?

## BREADTH MODEL INSTRUCTIONS

Green Hat directives:
1. Map the complete feedback structure. Identify all reinforcing and balancing loops. Name each loop and state what behavior it drives.
2. Identify system archetypes present — Fixes That Fail, Shifting the Burden, Limits to Growth, Eroding Goals, Escalation, Success to the Successful, Tragedy of the Commons, Growth and Underinvestment. State which applies and what behavior it predicts.
3. Identify leverage points ranked by depth using Meadows' hierarchy: parameters < buffer sizes < stock-and-flow structure < delays < feedback loops < information flows < rules < self-organization < goals < paradigms.

Yellow Hat directives:
1. Identify the highest-leverage interventions — structural changes producing the most beneficial system-wide effects.
2. Distinguish robust interventions (work across system states) from fragile interventions (depend on specific conditions).
3. Identify what the user gains from systems-level understanding — how seeing the feedback structure changes what they would do compared to linear analysis.

## EVALUATION CRITERIA

Extends the base four-criterion rubric with:

5. **Feedback Structure.** 5=all significant loops identified, classified (R/B), named, and behavioral implications stated. 3=major loops identified but one significant loop missing or misclassified. 1=loops not identified or confused with linear chains.
6. **Archetype Recognition.** 5=archetypes correctly identified with evidence, predicted behavior matches observed behavior. 3=archetype named but fit approximate. 1=no archetype recognition or archetype misidentified.
7. **Leverage Point Quality.** 5=leverage points ranked by depth with reasoning, highest-leverage interventions identified and justified. 3=leverage points identified but not ranked. 1=proposed interventions are parameter adjustments without structural analysis.

## CONTENT CONTRACT

The output is complete when it contains:

1. **System boundary** — what is inside the system and what is outside.
2. **Stocks and flows** — accumulations and rates of change defining the system's state.
3. **Feedback loops** — all significant loops classified as reinforcing or balancing, named, with behavioral implications.
4. **Delays** — where effects lag behind actions, and how delays shape behavior.
5. **System archetypes** — which known behavioral patterns are present, with evidence.
6. **Leverage points** — ranked by depth, with specific recommendations for highest-leverage interventions.
7. **Emergent behaviors** — counterintuitive or unintended consequences arising from structure.

## KNOWN FAILURE MODES

**The Everything-Connects Trap:** Unfalsifiable holism — claiming everything connects without analytical precision. Correction: Define the system boundary. Identify specific connections with specific mechanisms.

**The Deep Leverage Trap:** Overemphasizing paradigm shifts at the expense of actionable parameter adjustments. Correction: Rank by leverage AND feasibility. A workable parameter change may outperform an unreachable paradigm shift.

**The Qualitative-Only Trap:** Producing causal loop diagrams without assessing magnitudes, delays, or relative loop strengths. Correction: Assess which loops dominate under current conditions. A reinforcing loop with low throughput may matter less than a high-throughput balancing loop.

**The Observer Blindness Trap:** Analyzing the system as if the analyst is outside it. Correction: Identify how the analyst and user are part of the system.

## GUARD RAILS

**Solution Announcement Trigger (standing guard rail).** WHEN recommending an intervention, THEN verify it has been tested against the system's feedback structure. A recommendation ignoring feedback dynamics is a linear solution applied to a nonlinear system.

**Structure-over-events guard rail.** Focus on feedback loops, delays, stocks and flows — not individual events or actors. Events are symptoms; structure is the cause.

**Feasibility guard rail.** For every leverage point, assess whether it is actionable given the user's position and resources.

## TOOLS

Tier 1: CAF (identify all system factors), C&S (trace consequences across time horizons — delays are central), Challenge (test whether loops are genuine), RAD (recognize system type, divide if multiple systems conflated).

Tier 2: Engineering and Technical Analysis Module for technical systems. Political and Social Analysis Module for institutional or policy systems.

Enrichment frameworks: Meadows' 12 leverage points hierarchy. Senge's system archetypes for pattern recognition.

## TRANSITION SIGNALS

- IF the analysis reveals specific failure points requiring backward causal tracing → propose **Root Cause Analysis**.
- IF the feedback structure embeds institutional interests → propose **Cui Bono**.
- IF the user wants to explore multiple possible futures for the system → propose **Scenario Planning**.
- IF the user begins defining a deliverable → propose **Project Mode**.
- IF a foundational assumption worth questioning surfaces → propose **Paradigm Suspension**.
- IF relationships are primarily structural without significant temporal dynamics → propose **Relationship Mapping**.
